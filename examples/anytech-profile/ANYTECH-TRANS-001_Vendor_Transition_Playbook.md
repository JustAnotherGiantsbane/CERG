## 1. Purpose and Scope

AnyTech is a small team running a vendor-oversight model. Switching a provider — MSSP or MSP — is a high-risk event that can create weeks of detection and response gaps if not managed carefully. This playbook defines the phases, checklists, and validation gates for transitioning between providers.

This playbook covers transitions where one provider replaces another. It does not cover:
- Adding a second provider alongside the current one (co-management)
- Bringing a capability in-house (internalization)
- Ending a provider relationship without replacement (capability retirement)

Those scenarios require separate planning and are not covered here.

### 1.1 When This Playbook Activates

1. The IT Director makes a termination or transition decision (from the [Quarterly Review Kit](ANYTECH-REVIEW-001_Quarterly_Provider_Review_Kit.md) decision framework)
2. A provider gives notice of service termination, acquisition, or material change
3. Contract renegotiation fails and non-renewal is the backup option
4. A material breach (P1 SLA breach with impact, multiple significant breaches over 90 days) triggers transition evaluation

## 2. Provider Transition Types

| Type | Risk Level | Timeline | Notes |
|---|---|---|---|
| MSSP-to-MSSP | High | 60-90 days | Detection coverage gap risk. Requires parallel run. |
| MSP-to-MSP | Medium | 30-60 days | IT operations disruption risk. Parallel run recommended. |
| MSSP to in-house | Very High | 6-12 months | Requires hiring, tooling, and facility build-out. Not covered here. |
| Training vendor change | Low | 15-30 days | Low security risk. Primarily administrative. |

## 3. MSSP Transition Phases

### Phase 1: Decision and Planning (Weeks 1-2)

**Gate: IT Director approves transition. Risk register updated with transition risk.**

- [ ] IT Director documents the termination decision with rationale
- [ ] Transition risk added to the risk register as a material risk
- [ ] Timeline drafted with key milestones
- [ ] New MSSP evaluation completed (RFP, technical assessment, reference checks)
- [ ] New MSSP contract reviewed against the Provider RACI baseline
- [ ] Legal counsel reviews termination terms in the current MSSP contract
- [ ] Budget secured for parallel run period (both MSSPs active simultaneously)
- [ ] MSP informed of the upcoming MSSP change
- [ ] Communication plan drafted (internal, regulatory, customers if applicable)

**Gate output:** Transition plan document. Signed IT Director approval.

### Phase 2: New MSSP Onboarding (Weeks 3-5)

**Gate: New MSSP can receive and triage test alerts.**

- [ ] New MSSP receives the monitoring scope document (systems, environments, alert categories)
- [ ] Connectivity established — new MSSP SIEM receives log feeds
- [ ] Alert routing configured — new MSSP receives a copy of all production alerts
- [ ] Initial detection coverage map generated by new MSSP
- [ ] Gap analysis — new MSSP coverage vs. current MSSP coverage
- [ ] Alert quality baseline established (first 2 weeks of alerts reviewed in depth)
- [ ] Communication channels established (weekly call, escalation paths)
- [ ] Provider RACI updated for the dual-provider period
- [ ] Contact lists shared and tested

**Gate output:** New MSSP operational for monitoring. Parallel run begins.

### Phase 3: Parallel Run (Weeks 4-8)

**Gate: 4 weeks of parallel operations with no significant gaps.**

Both MSSPs operate simultaneously. New MSSP receives all alerts and builds operational maturity. Current MSSP remains the primary responder.

| Week | Activity | Validation |
|---|---|---|
| 1 | Both MSSPs monitor. Current MSSP is primary responder. | New MSSP alert quality scores against baseline. Coverage comparison. |
| 2 | Both MSSPs monitor. Current MSSP is primary responder. | Joint exercise — scenario run with new MSSP as observer. Maturity assessment. |
| 3 | Both MSSPs monitor. New MSSP begins triaging in shadow mode. | New MSSP triage quality scored vs. current MSSP benchmark. Differences documented. |
| 4 | Both MSSPs monitor. New MSSP participates in a real exercise as primary. | Exercise findings register. Handoff timing measured. |

**Gate output:** New MSSP meets or exceeds the current MSSP's alert quality score and coverage coverage. Gaps documented and accepted.

### Phase 4: Cutover (Week 8-9)

**Gate: New MSSP becomes primary responder. Current MSSP in monitoring-only mode.**

- [ ] New MSSP designated as primary responder in all documentation
- [ ] Current MSSP moved to monitoring-only mode (receives alerts but not primary)
- [ ] Provider RACI updated — new MSSP as D, old MSSP as shadow monitoring
- [ ] Communication updated — escalation paths, on-call contacts
- [ ] All internal documentation updated to reference new MSSP
- [ ] Current MSSP instructed to shadow only; no response actions

**Gate output:** New MSSP is primary. Current MSSP on shadow for 2 more weeks.

### Phase 5: Old MSSP Offboarding (Weeks 9-10)

**Gate: Current MSSP fully disconnected. Knowledge transfer complete.**

- [ ] Final incident and alert data pulled from current MSSP
- [ ] Detection rules and tuning documentation transferred
- [ ] Forensic artifacts returned (if any)
- [ ] Log feeds redirected from current MSSP
- [ ] Access revoked for current MSSP personnel
- [ ] Contract close-out completed
- [ ] Risk register updated — transition risk closed
- [ ] Lessons learned conducted

## 4. MSP Transition Phases

MSP transitions follow a similar but shorter pattern. The key difference: MSP transitions affect IT operations (patching, access, backups), not detection and response.

### Phase 1: Planning (Week 1)

- [ ] IT Director approves transition
- [ ] Transition risk added to risk register
- [ ] New MSP evaluated (scope, SLAs, security posture)
- [ ] Timeline drafted (30-60 days)

### Phase 2: Onboarding (Weeks 2-3)

- [ ] New MSP receives environment documentation (systems, accounts, contacts)
- [ ] Connectivity established — new MSP gains admin access to managed systems
- [ ] Initial patch compliance baseline established
- [ ] Communication channels established

### Phase 3: Parallel Run (Weeks 3-6) — Recommended

- [ ] Both MSPs have access. Current MSP remains primary.
- [ ] New MSP performs shadow patching and access management
- [ ] Weekly validation of new MSP's output quality
- [ ] Joint access review — compare MSP access lists

### Phase 4: Cutover (Week 6)

- [ ] New MSP takes over primary IT operations
- [ ] Current MSP access revoked
- [ ] Provider RACI updated

## 5. Parallel Run Procedures

### 5.1 Alert Routing During Parallel Run

During MSSP parallel run, all production alerts must be routed to BOTH MSSPs. Log sources must be configured to send to both SIEMs, or use a log-forwarder that duplicates to both destinations.

If dual routing is not technically possible:
1. Route all alerts to the primary (new) MSSP first
2. Configure the new MSSP to forward a copy to the current MSSP using their forwarding capability
3. Validate that the current MSSP receives all alerts with no more than 5-minute delay

### 5.2 Responsibility Matrix During Parallel Run

| Activity | Current MSSP | New MSSP |
|---|---|---|
| Alert monitoring (24/7) | Primary | Secondary (shadowing) |
| Triage | Primary | Shadow (score for quality, do not act) |
| Containment | Primary | Observe only |
| Incident command | Primary | Observe only |
| Reporting (SLA, metrics) | Primary | Provide shadow metrics for comparison |
| Joint exercises | Participant | Primary |
| Weekly quality review | With CERG | With CERG (separate) |

### 5.3 Transition Risk Register

Add the following risks during any provider transition:

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Detection gap during cutover | Medium | High | Parallel run ensures continuous coverage. Rollback available. |
| Alert routing failure | Medium | High | Test routing daily during parallel run. Have documented fallback. |
| Knowledge loss during offboarding | Medium | Medium | Mandatory knowledge transfer sessions. Documentation review. |
| New MSSP underperformance | Low | High | Do not cut over until parallel run validates quality meets or exceeds current provider. |
| Provider-to-provider handoff failure | Medium | Medium | Joint exercise during parallel run tests coordination directly. |

## 6. Knowledge Transfer Requirements

### 6.1 From Outgoing MSSP

- [ ] Detection rules inventory — all custom rules, their purpose, tuning history
- [ ] Alert tuning documentation — false positive exceptions, suppression rules
- [ ] Incident history — all declared incidents, post-incident reports, RCA documents
- [ ] Forensics artifacts — collected evidence for all incidents (where contractually obligated)
- [ ] Playbook documentation — any custom playbooks developed for AnyTech's environment
- [ ] System inventory — all systems monitored, monitoring configuration
- [ ] Contact lists — internal AnyTech contacts, MSP contacts, vendor contacts
- [ ] Exercise history — exercises conducted, findings, improvement item status
- [ ] Open improvement items — all open items and their status

### 6.2 To Incoming MSSP

- [ ] Monitoring scope document
- [ ] Current detection coverage map
- [ ] System inventory with criticality tiers
- [ ] Alert severity definitions and escalation criteria
- [ ] Contact lists and on-call rotations
- [ ] Current SLA targets and measurement methodology
- [ ] Provider RACI
- [ ] AnyTech's evidence quality expectations
- [ ] Joint exercise schedule for the first 12 months

## 7. Risk Management During Transition

The transition itself is a material risk that must be registered, tracked, and accepted.

### 7.1 Transition Risk Record

| Field | Value |
|---|---|
| Risk ID | RISK-TRANS-[YYYY]-[NNN] |
| Description | Transition from [current provider] to [new provider] for [capability, e.g., MSSP SOC/IR] |
| Inherent risk | High |
| Residual risk (with mitigations) | Medium |
| Transition period | [Start date] — [End date] |
| Risk owner | Engineering Pillar Lead |
| Acceptance authority | IT Director |
| Mitigations | Parallel run, daily routing tests, rollback plan documented |

### 7.2 Weekly Transition Status

During the transition, the Engineering Lead provides a weekly status to the IT Director:

1. Transition phase and progress against timeline
2. Validation scores — both providers compared
3. Any issues or gaps identified
4. Rollback readiness — could we revert if needed?
5. Go/no-go recommendation for next gate

## 8. Transition Rollback Criteria

The IT Director may order a rollback if any of these conditions are met:

1. **Detection coverage gap** — new MSSP coverage is below 90% of current MSSP coverage after week 2 of parallel run
2. **Alert quality below threshold** — new MSSP alert quality score is below 75% after week 3
3. **Critical routing failure** — any P1 or P2 alert not received by the new MSSP within SLA
4. **Provider coordination failure** — new MSSP and MSP are unable to coordinate on a shared system
5. **IT Director decision** — any material concern that erodes confidence in the transition

### Rollback Procedure

1. Instruct current provider to resume primary responder status immediately
2. Document the rollback trigger and rationale in the risk register
3. Conduct a lessons-learned session within 10 business days
4. Determine whether the transition can be retried with adjustments, or whether a different provider should be evaluated
5. The transition risk remains open until the IT Director closes it

## 9. Post-Transition Validation

After the transition is complete (old provider offboarded, new provider operating as primary), conduct validation at:

| Interval | Validation Activity |
|---|---|
| 30 days | Alert quality deep-dive — double the normal sampling rate (30 alerts/week) |
| 60 days | Joint exercise — coordinated scenario with new MSSP and MSP |
| 90 days | First quarterly review under new provider — standard scorecard |
| 180 days | Full evidence quality audit — all provider evidence categories |
